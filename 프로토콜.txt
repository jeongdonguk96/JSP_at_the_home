프로토콜
  통신을 위한 서로 간의 합의 혹은 규칙.
  주고 받을 데이터에 대한 형식을 정의한 것.
  
HTTP - html을 주고받는 프로토콜이라 생각하면 쉬움(마치 편지처럼).
  1. HTTP는 하이퍼(html) 텍스트를 전송하는 프로토콜. 단순하고 읽기 쉬움.
  2. 클라이언트의 정보를 저장하지 않음. 즉, 상태를 유지하지 않음.
     (요청을 연속으로 2번 보내도 같은 클라이언트라고 인식 못함)
  3. 확장 가능함. 커스텀 헤더 추가 가능.

응답메세지
  우리가 웹 주소창에 구글의 주소를 입력하면(=> 요청)
  그 주소가 프로토콜, 메서드, 호스트,커넥션 등의 형식이 적힌 메세지로 작성됨.
  그걸 기반으로 서버측에서 정보를 읽고 
  마찬가지로 응답시에 요청에 대한 응답 내용이 담긴 메세지를 보내줌

응답 메세지의 Error
  200번대 성공
  300번대 다른 URL요청
  400번대 클라이언트측 에러
  500번대 서버측 에러

요청메세지
  요청매서드는 크게 GET방식과 HOST방식이 있다.(외에도 있지만)
 
GET방식(READ) - 서버에서 리소스 얻어올 때
  바디에 내용이 없다. 대신 쿼리스트링에 내용을 적는다(소용량).
  URL에 데이터가 노출되기 때문에 보안에 취약
  데이터 공유(전달)에 유리

POST방식(WRITE) - 글쓰기, 로그인, 파일첨부 등등 서버에 데이터를 올릴 때
  바디에 내용이 있다(대용량).
  보안에 유리. 하지만볼 수 있기 때문에 암호화가 필요.(https = http + tls)
  데이터 공유에 불리.
